Introduction



The purpose of semantic data is to give meaning to data in a way that it is understandable by machines. Through this you can make relationships between data from any domains, and better yet, this potentiates the integration of systems into different databases. These relationships are possible with the paradigm called linked data, a set of norms that should be used to make semantic data.


Datasets


The databases for storing semantic data are called triple stores, this is because all data relating to something is stored in triple formats (Subject → predicate → value). In the figure to the side, each image is considered a triple store and the edges represent the related data links. Click here to view the community site for the pioneering triple data stores open project. Click in this link to open the website that shows the triple stores diagram of this project.


Ontology



Ontologies are the basis for linked data formation. An ontology is an abstraction of a concept. Where a concept has features that are assigned a set of instances. Its definition is made in a specific language so that it is structured in a way that is understandable by machines. Didatically we can make an analogy to a class in object orientation.


Vocabulary


One of the components of ontologies is vocabularies, these are directed to infinite specific domains like health, finance, commerce, social networks, science, etc ...Some examples of these vocabularies are FOAF, Schema.org, Vcard, Dublin Core, SKOS, among others. Each vocabulary holds a set (and growing) of properties / predicates. The predicates in turn are those responsible for establishing the domain that the property will store - similarly, we compare to the data members in a class, or, the names of the fields in the tables of a relational database. Currently o Linked Open Vocabularies (LOV) is the most popular site for accessing vocabulary information.


Resources


The defined ontologies allow the creation of resources and these are stored in triple stores. Maintaining the comparison the resources created are the instances of the objects, with identifier (in this case defined by a URI), defined vocabularies and predicates, and values assigned to them. The language used for resource definition is RDF/XML. The API teaching application presents this usage.




Finally we can realize that an ontology can contain as many vocabularies of different domains are needed to represent a data. Once the predicates are defined, how many are needed, they can receive values relative to their domain. These values can be a literal value (a name, a number, etc ...) or the URI of another resource. There is a specific notation for this representation in RDF/XML.


What is necessary to use this framewrok?

SemanticMake was developed in Java and has packages that have routines that encapsulate various natural operations to manipulate linked data, such as retrieval, update, creation and deletion of resources. To start using them you only need to add the semanticMake.jar container to your project and use its routines as needed. This can aid in the back-end development of your web application. It should be noted that saving, updating, and deleting a resource are limited to ontologies. There are four ways of representing a value in a predicate: a literal, a resource URI, a literal value in a subproperty, or a URI of a resource in a subproperty. However, if these possible forms are not sufficient to maintain the data types for the desired ontologies, the developer will now have more work to program methods that meet their needs, but will still be supported by a set of methods that meet the desired ontologies.

In addition, an API, called SemanticAPI, is available that enables the manipulation of linked data. On the server side this API was developed in Java and with the request methods use the REST architecture. The SemanticMake.jar container already contains this implementation. Its data manipulation methods are also limited to the ontology structures as well as the methods of the extensible packages. And these methods can not be increased. To use this API you need to import the sematicAPI.js library and this library has several calling functions, one for each of the API server methods and some utilitarian functions. Classes are available for configuring the application server address and the triple store address. If the triple store address is not passed then the API stores the data in a dataset making that service available and the database name is created and returned automatically the first time a data is persisted. The name of this database must contain a "workspace" + a sequence of 8 random digits, this name should be saved for later recordings.

In the testing tool the passed parameters must be variables or JSON objects in the formats expected by the functions of the API server module, as shown in the examples of the tool. In the other sections of the site you will find all documentation of both the packages and the API and some examples of how to use it. The figure below represents the SemanticMake schema.

                                    deleteGraph()
